# Trust Baseline Test Inventory - 2026-06-17

Status: Agent T2 evidence slice for `docs/AUTONOMOUS-BUILD-AUDIT-REMEDIATION-PLAN-2026-06-17.md`.

This is evidence taxonomy only. It does not introduce a domain-model term or a hard architectural decision, so no `CONTEXT.md` or ADR update is required.

## Label Meanings

- `default-entrypoint`: exercises a product/service entrypoint that the running product also uses.
- `opt-in-path`: proves behavior only after a non-default property, tag, profile, or lane is selected.
- `unit-only`: exercises an isolated class, helper, reducer, handler, or component.
- `static-only`: validates static artifact shape, SQL text, generated package metadata, or deployability heuristics without executing the runtime.
- `false-green-risk`: can go green while the default running product still does not use the new path.

- `[read]` `backend/src/main/resources/application-test.yml` uses H2, `ddl-auto=create-drop`, and `flyway.enabled=false`.

## Trust Baseline Conclusion

The named tests are useful regression guards, but none of them alone is completion proof for Builder, Chat, or Workspace activation. Completion evidence must pair a default product entrypoint with real seeded catalog data when migrations matter, and it must record whether the new path was actually reached.

For catalog/Flyway-dependent claims, prefer Postgres/Flyway reruns such as:

```bash
cd backend
DB_PORT=5433 ./gradlew backendIntegrationTest --tests 'com.pulse.config.PostgresFlywayMigrationSmokeIT' --tests 'com.pulse.config.V153V154PostgresCatalogIT'
DB_PORT=5433 ./gradlew backendIntegrationTest --tests 'com.pulse.pipeline.SchemaPropagationE2EIntegrationIT'
```

For default chat activation, the backend proof now runs without forcing `pulse.chat.orchestration=graph`, the default Postgres checkpointer is proven in the postgres-it lane, staging canvas decision behavior is pinned in Vitest, and a targeted running-app SSE proof verifies ChatPanel-to-StagingCanvas rendering. For Builder activation, the focused backend proof now asserts `CodeGenerationService.generate()` reached `CodegenOpEngine` for op-listed blueprints, the schema propagation proof asserts `SourceSQL` source-prepare/fallback behavior, and `CodeGenerationPostgresCatalogIT` proves the generate/package cutover against real Postgres/Flyway seeded catalog data. Live runtime execution remains a separate acceptance surface.
