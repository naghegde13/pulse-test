# SQL authoring ‚Äî SourceSQL + SqlModel blueprints over a deterministic Calcite validator, with inline date mnemonics

Status: accepted 2026-06-16 (operator + orchestrator). Builds on ADR 0011 (deterministic schema is the Builder contract), ADR 0012 (blueprint behavior = a closed set of 32 primitive ops), ADR 0013 (the Builder is a deterministic op-composition compiler; the LLM is Chat-side only; `sql-model` is the SQL escape hatch). **Refines ADR 0013 ¬ß3** (the "SQL-chaining" path) into a buildable shape.

## Context

ADR 0013 ¬ß3 named a power-user "SQL-chaining blueprint" ‚Äî a chain of `sql-model` ops carrying user SQL, with output schema derived by Calcite (the "Phase-2" prerequisite) or a declared schema. It did not pin: how the chain is shaped, where Calcite sits, how SQL reads from a relational *source*, how a free-form transform differs from a source read, or how PULSE's existing date-mnemonic facility reaches SQL. The operator grilled these to zero-fuzziness on 2026-06-15/16; this ADR records the locked decisions so `SPEC-calcite-sql-model.md` can be authored and gated.

Two distinct needs drive SQL authoring:
- **(a) In-pipeline free-form transforms** the 32 closed-vocab ops can't express (cross-source joins, arbitrary projections over already-ingested/derived data).
- **(b) Reading from a relational source via a SQL query** (source-side projection/filter/join, e.g. `SELECT ‚Ä¶ FROM orders WHERE biz_date = ‚Ä¶`).

The hard constraints are unchanged: schema stays **deterministic, zero-LLM** (ADR 0011, unknown ‚Üí loud fail); the Builder stays **deterministic** (ADR 0013); the op vocabulary stays **closed at 32** (ADR 0012); deployed pipelines run **without phoning home to PULSE** (PULSE is dev-only; enforced by `ForbiddenTokenScanner` INTERNAL_TOKENS at `ForbiddenTokenScanner.java:71-84` + `PULSE-MAP.md:248`).

## Decision

1. **Two SQL surfaces, NO new op.**
   - **`SqlModel` blueprint** (in-pipeline transform, need (a)) ‚Äî wraps the existing **`sql-model` op (op 27)**. Its input is **required**. The SQL is a **chain of discrete statements** `steps: [{ name, sql, materialize }]`; the **last step's `SELECT` is the output schema**. The op's input is referenced by the reserved name **`input`**; each step is referenced by its `name`; a step may reference `input` and any earlier step. `materialize: true|false` (default `false`) is a **performance hint** ‚Äî `true` persists that step's result as a temp/intermediate; it is **schema-irrelevant** (Calcite derives the same schema either way).
   - **`SourceSQL` blueprint** (relational source via SQL, need (b)) ‚Äî a new **INGESTION** blueprint built on the existing **`read-source` op** with a **`source_query`** param bound to a **JDBC connector**. The query executes at the source; **the source validates it** ‚Äî `SourceSQL` uses a **SIMPLE** SQL builder whose Validate button **round-trips to the bound JDBC source** (`PreparedStatement.getMetaData()` on a `SELECT ‚Ä¶ WHERE 1=0` / `LIMIT 0`), so the source DB validates its **own** dialect + functions and returns result-set metadata, which PULSE maps (JDBC types ‚Üí PULSE types) onto the output port. **No Calcite, no PULSE-side per-dialect parser** for `SourceSQL` (LOCKED 2026-06-16). `SourceSQL` feeds `SqlModel`.
   - **Neither adds an op.** ADR 0012's 32-op vocabulary is intact: `read-source` gains a `source_query` + source-validated (JDBC prepare/metadata) schema mode; `sql-model` is the in-pipeline transform. (`source_query` already exists on `BulkBackfill` ‚Äî V7/V81/V92 ‚Äî so the param shape has precedent.)

2. **`CALCITE-PHASE-2` = a deterministic SQL validator (zero LLM; ADR 0011/0013), `sql-model`-ONLY.** Given `(SQL, catalog)` it parses, validates, and **derives the result schema**, or **fails the build loudly**. It runs in **ONE catalog context** (LOCKED 2026-06-16): the **`sql-model`** upstream **`input`** schema plus the accumulated **prior-step** schemas. **`SourceSQL` is NOT Calcite-validated** ‚Äî it validates via the source (clause 1: JDBC prepare/metadata round-trip), so Calcite carries no discovered-source-catalog context. A **declare-schema fallback** applies when Calcite cannot parse a `sql-model` construct (and, for `SourceSQL`, when the source is unreachable at design time). Calcite runs at **build/design time and only computes schema** ‚Äî it **never executes** the SQL and **never resolves** a date value.

3. **`SourceSQL` is a dedicated, connector-bound blueprint** ‚Äî not a repurposed `BulkBackfill`, and not connector-free. Reading a real DB requires a connector + credentials (SecretRefs) + discovery + data movement; that machinery lives once in the Producer Registry + `read-source`, and `SourceSQL` **reuses** it. (`BulkBackfill`'s date-range/chunk/parallelism params are backfill-specific; overloading it would be wrong.)

4. **Date mnemonics work anywhere SQL or a SQL expression appears, via the inline token `[[ mnemonic ]]`** (e.g. `[[ PBD-1 ]]`). This **reuses** PULSE's existing facility ‚Äî the closed, 3-way-synced vocabulary (`DateMnemonic.java` ‚Üî `pulse_dates` ‚Üî `PulseSystemPrompt`), config-time validation (`DateMnemonic.validateOrThrow`), and runtime resolution (`pulse_dates.resolve_mnemonic`). The **Builder lowers** the token **per engine at build time**: for **dbt**, `[[ m ]]` ‚Üí `{{ var('pulse_<slug>') }}` in the model, and the Airflow task resolves `m` ‚Üí a date at runtime and passes it to the dbt step (`dbt build --select ‚Ä¶`, `CodeGenerationService.java:597`) via `--vars`; for **PySpark / `SourceSQL`**, the task resolves at runtime and string-substitutes the date into the SQL. **Calcite types `[[ ‚Ä¶ ]]` as `DATE`** for validation; it never resolves the value. Runtime resolution depends **only on Airflow** (`{{ ds }}` ‚Üí `PULSE_BUSINESS_DATE` env + the pinned **calendar bundle** (`calendar_bundle_uri`/`hash`) + Airflow Variables for time-dimension state) ‚Äî **never the PULSE DB** (no phone-home). The token `[[ ‚Ä¶ ]]` is chosen because the four existing conventions are taken or wrong-phase: `__UPPER__` (build-time codegen fill), `{{ ‚Ä¶ }}` (dbt Jinja + Airflow), `{name}` (PySpark f-string), `${‚Ä¶}` (flagged by `ForbiddenTokenScanner`). `[[ ‚Ä¶ ]]` is **consumed at build** (lowered to the engine-native slot), so nothing survives into the artifact and the scanner is unaffected.

## Consequences

- `CALCITE-PHASE-2` becomes a scoped build with its own gated spec, `docs/build-specs/SPEC-calcite-sql-model.md`. Until it ships, `sql-model` uses the **declare-schema** path (ADR 0013); once it ships, **Calcite-validate is primary, declare-schema is the fallback**, and the `CALCITE-PHASE-2` prerequisite flag is removed from #1 ¬ßB rule 27 and #2 ¬ßC.4.
- `read-source` gains a `source_query` + **source-validated** (JDBC prepare/metadata) schema mode ‚Äî **not** a Calcite mode. This **SIMPLIFIES** the validator: Calcite loses its second (discovered-source) catalog context and validates `sql-model` only; the source DB is the authority for `SourceSQL`'s own dialect/functions, so PULSE needs no per-dialect source parser. Two new catalog rows ‚Äî the `SqlModel` transform blueprint and the `SourceSQL` ingestion blueprint ‚Äî enter the catalog (#5) and the V153 seed (#2 ¬ßD).
- The inline-mnemonic token is a **cross-cutting** facility: #2 (codegen) emits the per-engine lowering; #4 (construct library / expression builder) inherits it; #1 rule 27 types it as `DATE`. The `ForbiddenTokenScanner` is unaffected (token consumed at build).
- The op vocabulary stays **closed at 32** (ADR 0012 intact).

## Considered

- **A new `sql-select` op for source SQL** ‚Äî rejected: a connector-bound source read belongs on the existing `read-source` op (credential/data-movement boundary), so a separate primitive would expand the closed vocabulary for no gain.
- **Calcite-validating `SourceSQL` against a discovered source catalog** (a second Calcite context) ‚Äî rejected (LOCKED 2026-06-16): the source DB is the authority on its **own** dialect + functions, and re-implementing per-dialect SQL parsing/typing in a PULSE-side Calcite catalog would be both lossy (Calcite ‚âÝ the actual source dialect) and a maintenance burden. Instead `SourceSQL`'s SIMPLE builder **round-trips to the source** (`PreparedStatement.getMetaData()` / `SELECT ‚Ä¶ WHERE 1=0` / `LIMIT 0`) and maps JDBC types ‚Üí PULSE types. This keeps Calcite **`sql-model`-only** (single catalog context) ‚Äî a net **simplification** of the validator.
- **`sql-model` with an optional connector** (one op spanning source + transform) ‚Äî rejected: it straddles the credential/data-movement boundary; connector-touching reads belong in `read-source` so "anything that touches secrets" stays in one op (clean security-review surface).
- **A connector-free `SqlSource`** (self-describing schema from SQL) ‚Äî rejected: the source-read case is already `SourceSQL`/ingestion; the only residual (a no-source "generate a date spine/seed table" case) is niche and is **out** (sql-model's input is required), foldable into `sql-model` later if a real blueprint needs it.
- **A single SQL statement with CTEs for the chain** ‚Äî rejected by the operator: discrete multi-statements are wanted so intermediate steps can be **temp-materialized** for performance.
- **`{{ ‚Ä¶ }}` or `${ ‚Ä¶ }}` as the mnemonic delimiter** ‚Äî rejected: they collide with dbt Jinja and with Spark `${}` / the `ForbiddenTokenScanner` respectively.

## Trade-off

The discrete-statement chain + per-step `materialize` + single-context (`sql-model`-only) Calcite + source-validated `SourceSQL` + per-engine mnemonic lowering is more emission surface than a single CTE statement and a single resolution path would be. **Accepted** ‚Äî it gives Data Engineers real performance control (materialization), a uniform inline-mnemonic experience across PySpark and dbt, and SQL reads from relational sources (validated by the source itself), all while keeping schema deterministic (ADR 0011), the Builder deterministic (ADR 0013), and the op vocabulary closed (ADR 0012).
